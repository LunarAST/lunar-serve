# lunar-serve

**Read-only HTTP distribution layer for LunarAST ecosystem data.**

`lunar-serve` provides a lightweight, zero-dependency HTTP interface for consuming `lunar-map.json` — the global topology generated by `lunar map`. It serves both human-readable Markdown summaries and structured JSON, with parameterized filtering for AI agents and dashboards.

---

## Quick Start

### 1. Generate Topology Data

```bash
lunar map -o /opt/lunar-map.json
```

### 2. Start the Server

```bash
lunar-serve /opt/lunar-map.json
```

Server listens on `http://0.0.0.0:8787` by default.  
To change the port: `export LUNAR_SERVE_PORT=8080`

### 3. Mirror Your First GitHub Project

Create a `repos.json` in the same directory as `lunar-map.json`:

```json
{
  "version": "0.5.0",
  "projects": [
    {
      "name": "my-service",
      "displayName": "My Service",
      "source": {
        "type": "github",
        "github": {
          "owner": "your-github-org",
          "repo": "my-service",
          "branch": "main"
        }
      },
      "visibility": "public"
    }
  ]
}
```

Restart `lunar-serve`, then your project is available at:

```
https://your-domain.com/your-github-org/my-service/tree/main
```

> **No `lunar scan` needed for basic mirroring** — the mirror displays whatever interfaces have been declared. To populate interfaces automatically, run `lunar scan` locally (for supported frameworks) or use AI to generate patches.

## Endpoints

### Ecosystem Summary

| Endpoint | Description |
|:---|:---|
| `GET /lunar-map.md?summary=true` | Ecosystem summary (~200 tokens, ideal for AI first contact) |
| `GET /lunar-map.md` | Full topology in Markdown |
| `GET /lunar-map.md?scope=<project>` | Single project scoped view |
| `GET /lunar-map.md?status=<orphaned\|unused>` | Filter alignments by status |
| `GET /lunar-map.md?path=<path>` | Filter alignments by path substring |
| `GET /lunar-map.md?style=mermaid` | Mermaid topology diagram |
| `GET /lunar-map.json` | Full topology JSON |

### GitHub Mirror URLs

Projects registered in `repos.json` with a `github` source type are accessible via URLs mirroring their GitHub paths:

```
GET /{owner}/{repo}/tree/{branch}
```

Example: `https://lunar.aifify.com/your-org/your-repo/tree/main`

This makes the URL immediately recognizable to both humans and AI agents.

### Unified API

All projects are also accessible via:

```
GET /api/v1/projects/{name}/map.md
GET /api/v1/projects/{name}/map.json
```

### Legacy Compatibility

```
GET /project/{name}
```

### Health Check

```
GET /healthz → 200 OK
```

## Adding a Project to the Mirror

1. **Edit `repos.json`** (placed next to `lunar-map.json`) and add a new entry to the `projects` array.  
   Choose `"type": "github"` for GitHub mirrors, or `"type": "local"` for local projects.

2. **Restart `lunar-serve`** to reload the configuration.

3. **The mirror URL is live** immediately. If the project has declared interfaces (via `lunar scan` or AI patches), they will appear in the Markdown snapshot.

> **Tip**: Combine this with `lunar patch` to easily populate interfaces from AI-generated YAML patches.

## Configuration

`lunar-serve` loads `repos.json` from the same directory as `lunar-map.json` to build GitHub mirror mappings and validate project identities.

### repos.json Format

```json
{
  "version": "0.5.0",
  "projects": [
    {
      "name": "cellrix",
      "displayName": "Cellrix",
      "source": {
        "type": "github",
        "github": {
          "owner": "your-github-org",
          "repo": "cellrix",
          "branch": "main"
        }
      },
      "visibility": "public"
    },
    {
      "name": "my-internal-service",
      "displayName": "Internal Service",
      "source": { "type": "local" },
      "visibility": "private"
    },
    "my-library"
  ]
}
```

Entries can be:
- **Simple string** (default: local project, public visibility)  
- **Detailed object** with `source` (github or local), `displayName`, and `visibility`

> **Privacy note**: Only projects with `"visibility": "public"` are accessible without authentication. Private projects require JWT (coming soon).

## Exposing to the Internet (Zero-Trust Security)

When running `lunar-serve` locally and exposing it to the public internet for AI Agent consumption, you MUST adhere to the following security baselines:

### Security Rules

1. **Default: Private projects are hidden on public endpoints**. Projects marked `"visibility": "private"` in `repos.json` are never exposed on public endpoints.
2. **Use `/private/` endpoints for sensitive data**. These endpoints require Ed25519-JWT authentication.
3. **Never expose raw HTTP endpoints without JWT for private data**. If using tunnels, ensure only public endpoints are exposed, or require JWT globally.

### Option 1: Cloudflare Tunnel (Free, Zero-config)

```bash
# Install cloudflared
curl -L https://github.com/cloudflare/cloudflared/releases/latest/download/cloudflared-linux-amd64 -o cloudflared
chmod +x cloudflared

# Start lunar-serve
lunar-serve /opt/lunar-map.json

# Create a temporary tunnel
./cloudflared tunnel --url http://localhost:8787
```

The terminal outputs a public URL like `https://lunar-serve-xxxx.trycloudflare.com`. Share this with your AI agent.

### Option 2: Tailscale Funnel (Permanent, Secure)

```bash
# Start lunar-serve
lunar-serve /opt/lunar-map.json

# Expose via Tailscale Funnel
tailscale funnel 8787
```

### Option 3: Ngrok

```bash
# Start lunar-serve
lunar-serve /opt/lunar-map.json

# Expose via Ngrok
ngrok http 8787
```

> **Important**: All tunnels only expose the data distribution layer. Your source code and local files are NEVER accessible through these tunnels. Only `lunar-map.json` data is served.

## Relationship with Other Components

| Component | Relationship |
|:---|:---|
| `lunar` CLI | Produces `lunar-map.json` consumed by lunar-serve |
| `lunar-scope` | Can consume `lunar-map.json` from lunar-serve in online mode |
| RouteAST Sub-Protocol | Defines the alignment statuses rendered in summaries |

`lunar-serve` is purely a read-only data distributor. It never writes to disk or modifies configuration files.
